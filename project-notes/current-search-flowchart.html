Current Actual Flow

Focamai Search Flow As It Is Running Now

This page describes the current implemented homepage flow, not the hoped-for flow. It is meant to make drift obvious in plain English.

Expected Fast Work

Validation, cache lookup, local ranking, and JSON formatting should normally be fast backend work.

Current Slow Work

OpenAI prompt generation and staged finalize scoring are currently in the critical path and dominate latency.

Rollback Route Still Exists

The old one-shot finalize route still exists, but the homepage now prefers the newer persisted staged flow.

1

User Starts Search

Homepage submit

The user enters a product topic on the homepage and clicks Start search. The frontend then launches two backend requests in parallel.

What the frontend does

Starts discovery and refine at the same time instead of waiting for one to finish first.

2

Discover Products

GET /api/search/discover

The backend validates the query, runs discovery, reads or writes guided cache, and returns preview results plus a discoveryToken.

Should be relatively normal backend work

Input checks, search retrieval, cache work, cleanup, and JSON response.

What the user gets

Preview candidates and enough saved context for later finalize steps.

3

Generate The Refine Prompt

GET /api/search/refine

The backend makes an OpenAI call to write the follow-up refinement prompt that asks what matters most.

Likely drift

This is currently a real LLM wait, not a tiny helper step. That is why refine is taking multiple seconds.

What the user sees

The AI refinement question, helper copy, and textarea placeholder.

4

User Asks For Focused Picks

Show focused picks

The user submits optional follow-up notes. The frontend does not call the old finalize route first. It starts a persisted finalize run.

Input sent

Query, discovery token, follow-up notes, retry count, and excluded candidate ids if this is a retry.

5

Start Persisted Finalize Run

POST /api/search/finalize/start

The backend rebuilds candidate context from the saved discovery token and creates a persisted finalize run.

What gets reconstructed

The backend rebuilds the candidate pool, applies retry exclusions, and stores run state for later polling.

Important distinction

This is not the old one-shot finalize route. It is the newer staged and persisted path.

6

Poll Finalize Status

GET /api/search/finalize/status

The frontend polls the persisted run. Each poll can advance part of the staged finalize work.

Why this can be slow

The request is not just reading status. It can actively move the staged ranking pipeline forward.

What the frontend is waiting for

Enough scored products for early reveal, then final settled shortlist state.

7

Run Staged Finalize Logic

Inside persisted finalize orchestration

The backend does staged ranking work, including AI-related scoring steps, before the shortlist is considered done.

Main concern

This is the part that appears to have drifted from “fast backend work” into “wait on more AI in the critical path.”

Current shape

Signals, rubric generation, shard scoring, and backend ranking/selection.

8

Optional Recovery Completion

POST /api/search/finalize/complete

If polling cannot finish cleanly after a run already exists, the frontend can ask the backend to complete the run deterministically.

Purpose

Recovery path after the persisted run has started, not the default primary path.

9

Show Final Shortlist

UI result state

Once the run is finalized, the homepage shows the six focused picks and the user sees the final shortlist.

What should feel fast

The UI reveal itself should be quick and calm once the shortlist data is already available.

What is hurting UX now

The user is waiting too long before the shortlist feels settled because backend finalize work is still expensive.

Your Intended Flow

Fast And Mostly Deterministic

This is the version you described in plain English: use AI where it helps, but keep most of the backend work local and fast.

1. Discovery fetches candidates

SerpApi returns the raw candidate pool and the backend cleans it down to the best 20.

2. Refine runs in parallel

AI writes the follow-up prompt and also prepares ranking guidance for later work.

3. User can already see early products

The user can view the first six deterministic picks immediately or continue with refinement.

4. Finalize sends context plus 20 candidates

The user adds more detail and the backend splits the pool into four shards of five.

5. AI scores shards

Each shard gets scored and ranked without turning finalize into a long multi-request workflow.

6. Backend picks the six best

Scores are merged locally and the backend chooses the final six products.

7. Optional AI polish on the final six

AI can label best match and secondary badges after the shortlist is already chosen.

Current Actual Flow

Slower And More Orchestrated

This is what the homepage is doing now based on the current implementation and the timing behavior you observed.

1. Discovery fetches candidates

SerpApi-backed discovery runs and returns preview results plus a discovery token.

2. Refine makes its own OpenAI call

Refine is currently a full separate AI wait for the follow-up prompt, which is costing seconds by itself.

3. User asks for focused picks

The frontend does not go straight into one compact finalize request. It starts a persisted run.

4. Start persisted finalize run

The backend rebuilds candidate context from the discovery token and stores run state.

5. Poll finalize status repeatedly

Status polling is not just reading state. It can actively advance staged finalize work.

6. Staged AI-related ranking runs inside finalize

Signals, rubric, shard work, and backend selection happen in the staged persisted flow.

7. Final shortlist settles later

The user waits for the persisted staged run to settle before the focused picks really feel done.

Biggest Drift: Refine

Your intended version uses refine as a lightweight parallel helper. The current version makes refine a real OpenAI wait that is already costing several seconds before finalize even begins.

Biggest Drift: Finalize

Your intended version sounds like one compact finalize pass with shard scoring and local backend selection. The current version is a persisted staged workflow with start, status polling, and in-progress backend advancement.

Where The Flow Seems To Have Drifted

  • The refine step became a real OpenAI wait instead of a lightweight helper.
  • The finalize step is not just fast local backend ranking. It now includes staged AI-related work.
  • The homepage is using the new persisted staged finalize path, not the old one-shot legacy finalize route.
  • The timing panel currently suggests that backend latency mostly means waiting on AI, not ordinary server work.
